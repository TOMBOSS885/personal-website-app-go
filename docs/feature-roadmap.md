# 个人网站后续功能优化方向

本文档用于记录 Go 后端 + React 前端个人网站后续可扩展的新功能方向。优先级以实用性、稳定性、安全性和实现收益综合排序。

## 功能方向总览

| 方向 | 功能 | 作用 | 优先级 |
|---|---|---|---|
| 访问体验 | 首页骨架屏 / 图片渐进加载 | 网络较慢时先显示占位内容，背景图、文章封面和项目截图加载更顺滑 | 高 |
| 访问体验 | Live2D 开关与低性能模式 | 移动端或弱性能设备可自动关闭 Live2D、复杂动画和部分装饰效果 | 高 |
| 内容管理 | 草稿箱 / 定时发布文章 | 后台写文章时可保存草稿，并设置未来时间自动发布 | 高 |

| 内容管理 | 文章标签管理页 | 统一新增、重命名、删除标签，避免文章标签越来越混乱 | 中 |
| 内容管理 | 图片资源库 | 统一管理文章图片、背景图、头像等资源，支持预览、删除和复用 | 高 |
| 音乐模块 | 音乐播放列表排序 | 后台拖拽调整歌曲顺序，前台按排序播放 | 中 |

| 站点安全 | 管理后台二次验证 | 可选邮箱验证码或 TOTP，提高后台登录安全性 | 中 |
| SEO | sitemap.xml 自动生成 | 自动输出站点地图，让搜索引擎更容易收录文章和项目页 | 高 |
| SEO | 文章 meta 信息配置 | 每篇文章可设置摘要、关键词、封面和 Open Graph 信息 | 中 |
| 前台互动 | 评论系统 | 支持文章评论，可先做管理员审核模式 | 中 |
| 前台互动 | 留言板 | 独立留言页，适合个人网站访客交流 | 中 |
| 数据统计 | 访问量统计 | 统计页面访问、文章阅读量、来源页面和访问趋势 | 高 |
| 数据统计 | 后台仪表盘增强 | 显示文章数、访问量、热门文章、最近操作和系统状态 | 中 |
| UI 个性化 | 主题预设管理 | 保存多套主题方案，一键切换亮色、暗色、图片背景方案 | 中 |
| UI 个性化 | 前台布局配置 | 后台调整首页模块顺序，如项目、文章、技能、音乐等 | 中 |
| 部署运维 | 后台健康状态页 | 显示数据库、上传目录、磁盘空间、API 状态和版本信息 | 高 |
| 部署运维 | 后台清理缓存 / 无用文件 | 清理未引用图片、旧背景、失败上传残留和临时文件 | 高 |
| 内容展示 | 项目详情页 | 每个项目独立展示截图、技术栈、介绍、链接和更新记录 | 中 |

| 多语言 | 后台多语言编辑增强 | 中英文内容并排编辑，缺失翻译时给出提示 | 中 |
| 移动端 | PWA 离线访问 | 手机可添加到桌面，缓存静态资源和部分文章 | 中 |

## 优先推荐实施

| 排名 | 功能 | 推荐原因 |
|---|---|---|
| 1 | 图片资源库 | 当前系统已经有背景图、文章图、头像、Live2D、音乐等上传能力，统一管理后会明显降低维护成本 |
| 2 | 登录失败限制 | 安全收益高，实现成本相对低，适合优先补齐 |
| 3 | 文章自动保存 + 草稿箱 | 后台写内容时体验提升明显，可减少误操作损失 |
| 4 | 全站搜索 | 内容变多后非常实用，文章、项目、技能都能快速定位 |
| 5 | 健康状态页 + 无用文件清理 | 对 Docker 部署、服务器排错和长期维护很有帮助 |

## 建议实施顺序

### 第一阶段：稳定性和安全

1. 登录失败限制。
2. 后台操作日志。
3. 后台健康状态页。
4. 无用文件扫描与清理。
5. 数据备份 / 导出。

### 第二阶段：内容管理效率

1. 图片资源库。
2. 文章自动保存。
3. 草稿箱。
4. 定时发布。
5. 标签管理页。

### 第三阶段：访问体验

1. 首页骨架屏。
2. 图片渐进加载。
3. Live2D 低性能模式。
4. 文章目录导航。
5. 全站搜索。

### 第四阶段：增强展示和互动

1. 项目详情页。
2. 访问量统计。
3. 后台仪表盘增强。
4. 评论系统。
5. 留言板。

## 可先落地的低风险功能

这些功能改动范围较小，适合作为下一轮优化起点：

| 功能 | 涉及范围 | 风险 |
|---|---|---|
| 登录失败限制 | 后端登录接口、数据库或内存限流 | 低 |
| 文章目录导航 | 前端文章详情页 | 低 |
| 音乐排序 | 后台音乐管理、音乐表排序字段 | 中 |
| 健康状态页 | 后端健康接口、后台新页面 | 低 |
| 图片资源库基础版 | 后台页面、现有上传目录扫描接口 | 中 |

## 后续选择建议

如果优先考虑服务器稳定和安全，建议先做登录失败限制、后台操作日志、健康状态页和无用文件清理。

如果优先考虑使用体验，建议先做图片资源库、文章自动保存、草稿箱和全站搜索。

如果优先考虑前台观感，建议先做首页骨架屏、图片渐进加载、文章目录导航和项目详情页。


整体看，这个项目现在已经不是“小玩具站”了，已经接近一个完整的个人站点系统：Go 后端、React 前端、后台管理、音乐、歌词、Live2D、Redis 缓存与限流、日志、安全面板、Docker 部署都有了。方向是对的，轻量化也比 Spring Boot 版本更适合你的服务器。

**当前优点**
- 后端已经 Go 化，接口结构比较清楚，部署体积和启动速度都不错。
- 前端做了代码分包，Live2D、后台、Markdown 等大模块不是一次性全塞首页。
- Redis 已经承担缓存、限流、访问统计、文章阅读量缓冲，能明显减轻 MySQL 压力。
- 上传限制、操作日志、安全日志、限流规则已经后台可配置，后期维护方便。
- Docker 构建会重新编译前端和后端，适合你现在的部署习惯。
- `/uploads/music` 被禁止静态访问，音乐走接口流式播放，这比直接暴露文件安全一些。

**最值得继续优化的方向**

| 方向 | 当前情况 | 建议优先级 |
|---|---|---|
| 性能 | 首页资源已经分包，但 `vendor-live2d` 和 `vendor-markdown` 仍较大 | 高 |
| 稳定性 | Redis 有降级，但 Redis 崩溃时本地限流 map 可能长期增长 | 高 |
| 安全 | 上传限制有了，但 Docker 当前 `network_mode: host` 隔离性一般 | 高 |
| 数据安全 | 有导出功能，但还缺完整的 MySQL + uploads 自动备份策略 | 高 |
| 部署 | 单容器跑 Nginx + Go API 简单，但后期扩展不如多容器清晰 | 中 |
| 观测 | 有日志后台，但缺实时性能指标、慢接口统计、错误率统计 | 中 |
| 前端体验 | 功能多了以后悬浮组件较多，移动端需要持续校验 | 中 |
| Live2D | 依赖本地 `l2d-widget`，懒加载已有，但体积仍大 | 中 |
| 音乐 | 已有限流和 Range 流，但音乐仍是最吃带宽的模块 | 高 |

**我最建议下一步做的 8 件事**

1. **给本地限流兜底加自动清理**
   Redis 不可用时会走 Go 内存限流。现在能用，但如果被大量 IP 扫描，内存 map 可能增长。建议加定时清理过期 key。

2. **增加结构化错误日志**
   现在后台能看安全/操作日志，但程序运行错误更多还是普通 log。建议统一成结构化日志，记录接口、耗时、IP、错误原因。

3. **增加慢接口统计**
   比如接口超过 `800ms` 就记录到数据库或日志，后台显示慢接口排行榜。这样以后你会很快知道是文章、音乐、Live2D 还是图片拖慢网站。

4. **生产环境关闭 AutoMigrate**
   首次部署可以 `AUTO_MIGRATE=true`，稳定后建议改成 `false`。后期数据库结构变更最好用明确的迁移脚本，避免旧库自动迁移出意外。

5. **给 uploads 和 MySQL 做自动备份**
   你的核心数据不只在 MySQL，也在 `uploads`。建议每天凌晨备份：
   - MySQL dump
   - uploads 压缩包
   - 保留最近 7 天、每周 1 份、每月 1 份

6. **进一步压缩图片和背景**
   后台上传背景、文章图片时继续强化 WebP/缩略图策略。访问时优先加载小图，必要时再加载大图，可以减少 50Mbps 带宽压力。

7. **音乐做更细的带宽控制**
   可以增加：
   - 单 IP 每日音乐流量统计
   - 单曲异常访问告警
   - 可选音频压缩版本，比如后台上传原文件，前台播放 128kbps/192kbps 版本

8. **Docker 部署拆分可选方案**
   当前一体镜像简单好用。以后访问量上来，可以拆成：
   - `frontend-nginx`
   - `go-api`
   - `redis`
   - 外部 MySQL  
   这样更容易单独重启、扩容和排查。

**安全上还要注意**
- `JWT_SECRET`、`ADMIN_PASSWORD`、`MEDIA_SIGN_SECRET` 必须强随机。
- 宝塔面板不要暴露数据库端口到公网。
- Redis 不要开放公网，只允许本机或内网访问。
- Docker 的 `network_mode: host` 虽然方便连宝塔 MySQL，但隔离性差，后期建议换 bridge 网络或明确防火墙规则。
- 上传文件夹不能允许执行脚本，你现在已经禁止 Live2D 上传脚本类文件，这点是对的。

**我的判断**
现在项目整体已经比较完整，最大问题不是“功能不够”，而是功能越来越多以后，要把三件事继续做好：**带宽控制、日志清理、异常可观测**。

如果只选一个最高收益优化，我会先做：**本地限流清理 + 慢接口统计 + 自动备份**。这三个对稳定性最实在。
